# Agent Harness Core Invariants

Date: 2026-06-30

This catalog is the release-gate source for deterministic simulation, scenario replay, and review evidence. The same invariant IDs are exposed by `agent-harness invariants` through `agent_harness_core::quality::invariant_catalog`.

Release rule: a regression that violates I1-I13 blocks cutover even if feature tests pass. I8 now includes `/new` as a prompt/memory task boundary, so fresh-task requests must not inherit previous-task imported memory context unless a future policy explicitly opts in. I9 now has focused completion-repair and stdout-recovery regression gates; changes that touch compact fallback, fresh-thread recovery, provider delivery, or image/media tool completion still need the broader final-surface replay matrix before promotion. I10 is an initial guard, not full tool-supervisor parity; future work should add per-tool cancellation and resumable execution before claiming the gap closed. I11 is deliberately generic: image-only compact or image-only prompt redaction is not enough when audio, browser, document, worker, provider-native media, generated media previews, Discord text/image attachments, or large tool-log artifacts can enter durable context. I12 is the bridge ownership guard: configured openclaw-mem write/recall must be primary, recall fallback must stay read-only, and store fallback must fail closed. I13 is the context-rollover guard: official compact success must either advance rollover accounting or leave a structured skip receipt before long-context continuity can be considered reliable. Codex/Node service executable and startup changes must prove spawn-stable native resolution and terminal startup/write failure receipts before cutover. Any new queue, tool, memory, or background-work path must either reuse these invariants or add a new invariant before release.
