# Agent Harness Topology Contract

Date: 2026-07-19

This contract turns the architecture handoff into release-gate input. Use it before changing channel ingress, channel state, prompt assembly, runtime queueing, Codex execution, final outbox, delivery, progress, workers, cron, or memory. The operations handbook remains the live operational source of truth; this file is the compact dependency map and scenario matrix that reviewers and external coding agents should apply to code changes.

## Identity Axes

Every channel-origin turn is keyed by these axes. A change that drops, normalizes, or compares only a subset of them can create cross-agent or cross-session regressions.

| Axis | Meaning | Must Be Preserved Through |
|---|---|---|
| `platform` | Telegram, Discord, or another channel adapter. | ingress, queue, progress, outbox, delivery receipts, trace |
| `accountId` | Provider account or bot identity when configured. | channel identity, outbox, delivery, receipts |
| `channelId` | DM/chat/channel/thread address. | channel state, queue item, delivery intent, trace |
| `userId` | Sender identity inside the channel. | permission gate, channel state, session key, trace |
| `agentId` | Runtime owner, not just persona text. | provider/model defaults, Codex home, prompt files, memory namespace, runtime lane, session freshness, outbox, delivery, OperationPlan scope |
| `sessionKey` | Conversation/session continuity key. | command state, runtime queue, Codex plan, transcript, freshness guard, prompt/memory task boundary |
| `virtualSessionId` | Stable task-continuity owner across concrete-session rollover. | working-context resolver, prompt manifest, operation plan, child result owner, coordinator resume |
| `runtimeClass` | Interactive, cron, worker, maintenance, or another runtime lane. | queue selection, lease, retry/dead-letter policy, status |
| `sourceQueueId` | Correlation for one queued turn. | Codex completion, final outbox, delivery receipts, trace |
| `backendContextGeneration` | Opaque Codex/backend context generation, not a routing wildcard. | prompt manifest, injection ledger, reinjection decision, prepared execution receipt |

Two digests serve different boundaries. The full concrete-lane identity includes `sessionKey` and `backendContextGeneration`; it must change across concrete rollover. The virtual exact-lane digest retains platform/account/channel/user/agent/runtime/root-virtual-session axes but deliberately excludes the concrete backend generation, so one `virtualSessionId` can own a frozen skill manifest across rollover without becoming a cross-lane wildcard.

## Mandatory Impact Matrix

Codex backend provenance is a deployment boundary. Candidate and live supervisors must resolve one deployment-owned canonical executable, never PATH or global npm, and must fail closed unless an `agent-harness.codex-backend-provenance.v1` receipt proves exact version `0.144.5`, executable SHA-256, npm/native package integrity, provider-scoped `CODEX_HOME`, config digest, and correlated parent/child lifecycle. Source and staging may prepare this receipt; live mutation still requires explicit operator cutover approval.

Backend authentication is a provider and operator boundary. Every provider resolves a canonical harness-owned `CODEX_HOME`; a newly created home is Unix `0700` or a protected current-operator-only Windows DACL, while existing-home ACL migration requires stopped-state approval. An empty home is explicitly unauthenticated and never falls back to a user-global Codex home. Only the interactive `backend-auth` operator command may run browser, device-code, API-key-stdin, cancel, refresh, or logout flows. Durable state and ordinary Telegram/Discord turns may carry only secret-free lifecycle/readiness/correlation evidence. Raw operation/login ids are reduced to one-way digests, and runtime-ready requires same-generation `account/read` plus `model/list` evidence linked to the canonical executable provenance receipt. With the default-off runtime gate enabled, a turn without ready evidence becomes nonterminal `auth-deferred`: it is non-claimable, consumes no retry budget, produces no final/error outbox, and is not given credentials or silently routed through another provider home. Only a strictly newer ready generation may append its deterministic exactly-once `retry-pending` wake; append-before-mark ordering makes restart reconciliation lossless.

Web search is a provider capability and exact-lane tool-authority boundary. The runtime reads `modelProvider/capabilities/read` on the same app-server connection, sends an explicit top-level `web_search` on thread start/resume, and never derives the mode from sandbox/full-access state. Cached is the ordinary default; live requires explicit freshness intent plus allowed lane policy; sensitive, offline, replay, missing-capability, or denied lanes are disabled. A thread may resume only when its receipted effective mode/provider/lane binding still matches. Search items remain untrusted turn context: durable action receipts hash query text, omit private URLs, and cannot themselves authorize memory, skill-learning, dream, command, or delivery effects.

## Agent-Boundary Scenario Pack

Run or add tests equivalent to these scenarios for channel/runtime/session changes:

1. Same agent, same channel/user, old session completes after `/new`: final reply is suppressed and an explicit stale-session warning is recorded.
2. Different agents, same platform/channel/user, one agent has an active session in shared channel state: the other agent's completed turn is not suppressed solely because the shared state belongs to a different `agentId`.
3. Non-main agent normal message reaches prompt assembly with that `agentId`, its own workspace/prompt files, provider/model defaults, Codex home, and memory namespace.
4. Completed turn produces exactly one source-correlated final outbox record or one terminal error/dead-letter notification.
5. Trace reconstruction can connect ingress, queue id, Codex completion, final outbox, and delivery or terminal notification.
6. Fresh request after `/new` uses the new concrete session as a task boundary: previous-task imported memory context is not injected unless a later policy explicitly opts in.
7. `/think max` and `/reasoning max` are command-only aliases over one agent-scoped state; the next normal turn carries exact `max`, while a sibling agent sharing platform/account/channel/user does not inherit it. Exact `ultra` is rejected and `ultra-high` remains the legacy `xhigh` alias.
8. Prompt manifest and injection ledger bind to the exact full lane plus backend context generation. A sibling agent or different lane cannot reuse another agent's injected configuration, and a changed/deleted config file or changed backend generation produces the expected bounded reinjection/tombstone behavior.
9. Sibling children may use different immutable provider/model/effort policies. Retry cannot silently change a child's policy, and every expected terminal child contributes either a bounded/redacted result with opaque pointers or an explicit failed-omission envelope to the exact parent owner.
10. Child/external-agent progress, completion, and error are internal evidence. The exact master coordinator consumes only validated envelope fields and structural metadata, waits until the whole exact lane is idle, schedules one idempotent continuation, and is the only participant allowed to emit parent user-facing output.
11. Skill F1 shadow alone writes one privacy-safe exact-lane route receipt and no manifest, delivery, usage, learning, or proposal state; F2 observation is separately enabled and still leaves active selection and prompt bytes identical.
12. A concrete backend rollover rehydrates the frozen virtual-session skill revision without a fresh route link or positive-use event; `/new` closes the old manifest and begins with no inherited task intent or active skill state.
13. Runtime completion records ordinary model final as unknown until external verification, joins delivery/outcome/episode under the same virtual session, emits one retry-idempotent terminal review, and produces only a reversible unapplied improvement proposal.
14. An ordinary final that needs more than 16 semantic presentation blocks is durably retained as canonical text without a lossy automatic presentation. A persisted legacy 16-block projection falls back before the first rich unit; Discord and Telegram deliver all ordered chunk bodies, failed required chunks report `fullTextPreserved=false`, and a terminal receipt suppresses restart replay.

The regression test `runtime_pipeline::tests::channel_session_freshness_does_not_cross_suppress_other_agent` covers scenario 2. The regression test `prompt::tests::prompt_bundle_new_command_boundary_skips_prior_task_memory_context` covers scenario 6 for prompt assembly.

## Support-Plane Knowledge Graphs

The topology contract is canonical text. Generated graph tools are support-plane aids: they help reviewers discover relationships, but they do not replace source code, tests, release gates, or openclaw-mem Store/Pack/Observe ownership.

Understand-Anything is a good fit for codebase topology exploration because it can generate `.understand-anything/knowledge-graph.json` with file/function/class/dependency nodes, architecture layers, guided tours, semantic search, and diff-impact views. Use it for these bounded jobs:

- Build or refresh a repo-local code topology graph after behavior-changing architecture work.
- Ask code-graph questions such as "which modules touch session freshness?" or "what changes affect final outbox delivery?"
- Compare the generated graph against this contract to find missing component edges or stale docs.
- Produce reviewer-facing navigation aids for large changes before release review.

Do not use it for these jobs:

- Do not treat generated graph output as the canonical memory store.
- Do not let it write live harness state, secrets, channel ledgers, or `.agent-harness` private data into tracked documents.
- Do not enable auto-update or dashboard launch as a release gate until the graph artifacts, ignore policy, and public hygiene behavior are verified.
- Do not merge generated graph facts into openclaw-mem as trusted facts without provenance, freshness, and human-readable source file pointers.

Recommended staged adoption:

1. Create `.understand-anything/.understandignore` that excludes `.agent-harness/`, `.debug/`, `.tmp/`, `target/`, `.public-export/`, secrets, generated images, and large ledgers.
2. Run a scoped graph build for tracked source and docs only.
3. Save the graph under `.understand-anything/knowledge-graph.json` as ignored local support state unless a separate public export policy is approved.
4. Record a compact memory/graph freshness receipt that links the graph commit hash, topology contract version, and source docs used for generation.
5. Use the graph to support impact review, while keeping `docs/agent-harness-topology-contract.md`, `docs/invariants.md`, and tests as the release gates.

`docs/topology-explorer.html` is the lightweight human-facing companion view for this contract. Generate it from the public topology source by running:

```powershell
python tools\generate_topology_explorer.py
```

The generated page embeds `docs/topology-explorer-data.json` so it can be opened directly from disk. It may help operators browse components, identity axes, release gates, and open gaps, but it is still a support-plane artifact; do not edit the HTML/JSON by hand or treat it as canon.

`agent-harness scenario-matrix` prints the executable release-gate catalog for topology-sensitive changes. Use it with `agent-harness invariants`, `agent-harness schema-registry`, and this contract when a diff touches channel identity/state, runtime queue/pipeline, Codex recovery, progress, final outbox/delivery, memory, or multi-agent ownership.

### Round16-1 Progress Ordering Binding

- `progress-delivery-volume-gap`: progress delivery preemption is queue-local. A global wake from another lane must not drop the whole pending batch, first-surface sends are not preempted before any provider message exists, and same-queue providerless orphan surface claims are reclaimed without waiting for the claim TTL.
- `runtime-terminal-control-gap`: progress delivery must consume durable terminal-control evidence before rendering non-terminal cached progress. Terminal-controlled queues may emit one terminal close edit for existing provider surfaces, record `progressSuppressedReason=terminal-control-present`, drop cached ghost events only after the close receipt is recorded, and then stay silent. Normal terminal progress paths keep their compacted terminal context so late events and retryable lane failures do not suppress legitimate retries.

### Battle-Set Progress Source-Authority Binding

- A last-committed or unavailable progress index snapshot is diagnostic evidence only. It must not create, edit, or replay provider-visible progress from `compactedEvents`; the bounded cache is retained until a fresh source projection can rebuild the plan.
- A fresh projection may contain retained history after a reset or migration. If a queue has no existing provider surface and its latest event is older than the progress freshness window, it is suppressed rather than creating a late `Send`; an existing provider surface may still converge through its normal edit/terminal path.
- `progress-delivery-loop` owns a watched stop file. It must check before provider I/O and release every unattempted fresh-send claim when stopping, so restart/cutover cannot strand a short-lived claim or require a broad process kill.

### Battle-Set Controlled Coordinator Replay Binding

- A provider-visible coordinator replay is an operator-only verification surface, not a general subagent import or recovery command. It must require an explicit bounded source marker, exact full-lane identity for `main`, its currently active exact channel session when v2 state exists, authoritative catalog admission for that lane and every planned child execution agent, exactly two distinct child models with at least one exact `max`, and a separate explicit confirmation.
- The command must use the existing V5 transactional child/wait/watchdog path. It must not create child work through direct database edits, generic historical-ledger selection, or an untyped runtime queue append.
- Child turns remain worker-origin internal work. The only provider-visible final is the exact master continuation after the mailbox and durable coordinator contract completes; failure evidence is retained privately and never retried by a scheduled self-check.

## Skill Ecosystem V2 Ownership And Activation

Skill serving and skill learning use three explicit lifetimes. A concrete `sessionKey + backendGeneration` owns prompt delivery and cache/rehydration state only. An exact-lane `virtualSessionId` owns bounded task intent, the frozen catalog/topology/skill revision manifest, attributable outcomes, and one terminal learning review. The agent library identified by `agentId + libraryRevision` owns canonical packages, topology, statistics, proposals, dream runs, and later activation epochs.

The joined evidence chain is `skill-routing.v2 -> skill-delivery.v2 -> skill-outcome.v1 -> virtual-session episode -> knowledge-classification.v1 -> skill-learning.v2 -> skill-dream-run.v1 -> catalog/topology activation`. Every link retains the same agent, root-session hash, exact-lane digest, and virtual-session ID. A concrete-session rollover may emit `deliveryReason=rehydration` for a frozen skill body, but it creates no fresh routing, use, outcome, or improvement event. `/new` terminalizes the old manifest and starts with no inherited task intent or active skill entry.

F1 shadow routing runs only after runtime preparation has the exact account-aware full lane. Its query handoff is internal and non-serialized; the receipt records current-task bytes, excluded ambient-note bytes, active v4 serving IDs, shadow candidate-card IDs, hard-gate rejections, catalog digest, and dormant topology revision without raw task text. Receipt failure or missing exact identity may add an operator warning but cannot change active selection, prompt bytes, usage priors, delivery, learning targets, or final output. A deterministic receipt ID makes repeated preparation overwrite the same private evidence record instead of manufacturing a second logical decision.

F2 virtual-manifest observation has its own default-off switch and requires the F1 route receipt; enabling F1 alone still writes no manifest or delivery evidence. The observer freezes task-intent/catalog/topology/skill revisions under the exact virtual-lane digest, appends deterministic `skill-delivery.v2` evidence, and leaves active serving bytes unchanged. F3 runtime capture records ordinary model final as unknown until externally verified, joins delivery/outcome/episode evidence, writes one retry-idempotent terminal review, classifies bounded admitted knowledge, and can emit only checksummed reversible unapplied improvement proposals. CLI synthesis is proposal-only unless `--apply` is explicit; worker apply requires both `applyAuthorized=true` and `proposeOnly=false`.

Replay evidence separates receipt-derived automatic metrics from reviewed routing labels. Every frozen manifest owns all required evaluation classes through either a seed fixture or an explicit expansion-plan owner, fixture bytes and counts are checksum-bound, and `private-local` cases require an explicit load mode. Only cases marked `routingLabelsReviewed` enter precision and forbidden-rate denominators; all 53 seed turns may still contribute selection, prompt-byte, duplicate-source, class, and lane metrics.

Agent-library maintenance has two ownership levels: parallel per-virtual-session extraction claims and one serialized `agentId + libraryRevision` consolidation writer lease with heartbeat. Scheduled, startup-catch-up, and operator triggers enter the same durable job ledger and cannot bypass source admission, budgets, replay, or proposal-only policy. Foreground work wins lease contention. Candidate artifacts are written only to a staged workspace, validated before watermark advancement, and a semantically unchanged workspace completes without a model review.

Topology is a versioned routing artifact containing controlled facets, typed edges, aliases/tombstones, and materialized exact-scope views. Normal reclassification does not move canonical skill packages. An active virtual session freezes its catalog/topology revisions; candidate policy or topology is exposed only through an exact cohort overlay. Activation requires a later catalog epoch, independent candidate and rollback checksums, held-out replay, and successful downgrade/restore rehearsal. Human dream/rebaseline reports are private, marked non-indexable, and rejected from matcher, memory admission, episode synthesis, and later dream inputs.

This architecture is dormant until separately qualified. Source/staging implementation and a cutover handoff do not authorize a live C1 or C2 activation. C1 installs backward-compatible shadow foundations while retaining current serving semantics; C2 separately activates serving policy and then a reviewed catalog/topology epoch for exact cohorts. Autonomous body or structural mutation remains a later optional authorization.
