## Summary

The harness is ready for controlled live Telegram/Discord handoff testing and basic agent turns through the Codex backend. The current upgrade also implements the OpenRouter provider path needed for future OpenRouter-backed agents, adds per-agent memory namespaces so `main`, `小小梨`, and other agents can keep independent memory/canvas state without leaking provider config across agents, and closes the previously deferred channel identity, delivery intent, runtime backoff, and cron scheduler items.

No feature parity item is left in a vague intermediate state. Harness-owned work is marked `Implemented`. Items that require a user decision, external service contract, live credential, trust policy, or Windows service registration are marked `Blocked` or `Action Required` with the exact next action. Items the user wants but has explicitly chosen to do later are marked `Deferred`.

Runtime loop concurrency currently uses a bounded OS thread scheduler instead of a Tokio runtime. The hot path remains long-running blocking Codex child processes, file leases, JSONL appends, and synchronous HTTP delivery; moving only this layer to Tokio would still require `spawn_blocking` for the runtime work while increasing the migration surface. Tokio becomes a better fit when HTTP clients, queue scans, outbox delivery, and runtime orchestration are converted together to async APIs.

Forward roadmap integration is now tracked as staging implementation status plus remaining live/soak gates. The 2026-06-12 external roadmap has been incorporated into `docs/agent-harness-core-roadmap-backlog.md` with implementation steps, acceptance standards, and a coverage matrix for every phase and track. The staging pass implements the directly codeable P0-P7/T/M/P6/P7 mechanisms: fail-closed config validation, retry/dead-letter, log rotation, queue retry/skip, supervision evaluation, health/trace/metrics, canary receipts, queue shadow compare, scoped stop, admission control, background registry, token reports, learning proposals, MCP request handling, encrypted repo-local vault, security scans, invariants, schema registry, changelog/release discipline, and CLI gates. Long-running live evidence remains pending where the review target requires real traffic or OS persistence.

The backlog acceptance standards are aligned with the external comparative review dimensions: concurrency, persistence, error recovery, supervision, security, observability, token efficiency, extensibility, testing quality, and maturity. A backlog item is not considered production-complete until its evidence matches the review target: automated tests are enough for deterministic code paths, while queue shadow parity, live channel smoke, reboot recovery, canary deploy, advisory audit, and 30-day SLO claims require live/soak receipts.

### Roadmap Notes and Corrections

1. Accepted correction: round3-2 B1 must not introduce `background-jobs.jsonl` as another execution store. Background lifecycle state goes into SQLite WorkerStore; JSONL is audit/receipt only.
2. Native MCP is adopted, but first implementation should be a minimal supported MCP subset with initialize/list/call protocol fixtures rather than every optional MCP feature.
3. Warm app-server pooling is gated by measurement and must prove no cross-session, cwd, auth, or agent-context contamination.
4. Vaulted secrets direction is decided: repo-local encrypted vault for cross-platform portability.
5. Canary deployment must support fake-only canary when live provider credentials are unavailable; live small-model canary is a stronger optional gate.
6. `/healthz` must default to local/admin-gated access and never expose secrets or raw prompt/memory content.

## Assessment

Harness-owned feature parity work is implemented for the current scope, including provider-isolated OpenRouter routing, multi-agent independent memory namespaces, channel identity binding, delivery intent, runtime backoff hints, cron scheduler watermarks, the OpenClawMem snapshot/service-adapter bridge, and the repo-local encrypted vault primitives. Non-implemented work is either gated by plugin trust implementation, live vault migration/rotation, Windows service-wrapper implementation, optional remote `openclaw-mem` service contract/credentials, or operator live-smoke actions, or explicitly deferred by user decision.
